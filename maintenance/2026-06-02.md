## Inspection Scope

Second routine-maintenance cycle, full-codebase. Follows the completion of
Mission 02 (Resilient Queue-Backed Playback — the two-engine playlist system:
a native Sonos queue path that survives MCP process reap, plus a worker-thread
fallback). The first cycle (2026-05-18) resolved 22 items via the Baseline
Maintenance mission; this cycle re-inspected everything plus the carry-forward
debt that Mission 02's debriefs flagged.

**Delegation plan executed**: Single wave, three concurrent agents — an
Inspector (source + tools; Categories 1, 3, 4, 5, 6, 7, 9), a Test Systems
Inspector (Category 2 + test-quality emphasis), and a Security Reviewer (deep
manual). Project is small (~2,090 lines across 8 source modules + ~1,740 lines
of tests + 4 root smoke scripts); no partitioning needed. An Architect (Opus)
assessed severity and verified findings against the live tree; no roundtable
cycle was required — the one cross-reviewer disagreement (transitive CVE
severity) was a downgrade both parties' evidence already agreed on.

**User-directed emphasis this cycle**: code deduplication, redundant/incorrect
comments, and reducing overlapping tests — alongside the debrief-flagged debt
(SoundHelix smoke dependency, doc-drift, coverage gaps).

## Executive Summary

mcp-sonos remains a healthy, well-maintained single-developer project. The
security posture is clean: all 2026-05-18 hardening (play_file confinement,
Piper voice hash pinning, http/https scheme allow-list, public-repo
anonymization) is intact, and the new Mission-02 surfaces (bulk queue enqueue
via DIDL, the stateless control surface) introduce no injection, command-
execution, or SSRF risk beyond the documented LAN-trusted threat model. All
four test-coverage gaps carried from Mission 02's debriefs are now closed. The
"Maintenance Required" verdict rests entirely on the three quality axes the
maintainer prioritized — a small cluster of code-duplication, redundant-comment,
and overlapping-test items, plus one cheap-but-irreversible hygiene gap (`.env`
not gitignored on a public repo) and the debrief-deferred SoundHelix smoke
single-point-of-failure. Nothing blocks further work; a single small
consolidation mission clears the actionable set.

| # | Finding | Severity | New/Known | Recommendation |

**Baseline hardening verified intact** (do not re-litigate): F2 `play_file`
confinement — `controller.py:205-221` resolves with `.resolve()` *before* the
`is_relative_to(media_root)` containment check, enforces an audio-extension
allow-list and `is_file()`; directory listing 404s at `audio_host.py:78-80`.
F8 Piper pinning — `KNOWN_VOICE_HASHES` + `.suspect` quarantine on mismatch +
verify-on-existing + trust-on-first-use logging, hash checked on the `.part`
file before atomic rename. F14 scheme allow-list — applied at the Pydantic
`AfterValidator` (`server.py`), controller `play_url`, and manager `add`/
`add_many`; the DIDL enqueue path only takes already-validated items.

**This cycle's concerns cleared**: no command injection (Piper runs in-process,
no subprocess/Popen/os.system anywhere); no DIDL/XML injection (SoCo serializes
title/uri via ElementTree `SubElement`, which escapes metacharacters; URI is
additionally `quote()`-encoded at `playlists.py:362`); SSRF bounded by the
http/https allow-list (the Sonos device fetches, not the MCP host).
`PIPER_VOICE` flows unvalidated into a filesystem path + HF URL but is
operator-set env config, not agent-reachable (`say`'s `lang` param is ignored).

**Detail (S-1)**: `git check-ignore .env` returns nothing; `.gitignore` has no
`.env` entry. Not directly exploitable, but if the operator copies
`.env.example` → `.env` and fills in real LAN IPs (`SONOS_IPS`/`HOST_IP`), a
`git add -A` could commit real internal topology to this public repo. No
credentials live in this project's config (IPs/ports/voice names only), so the
impact is LAN-topology disclosure — but a publish you cannot un-publish,
prevented by one line, warrants Action Required regardless of low impact.

## Recommendations

1. **Land the single maintenance mission.** All six Action-Required items are
   low-risk and behavior-preserving, backed by the existing 63-test suite. The
   folded-in Advisories cost almost nothing extra.
2. **Do S-1 first.** A `.gitignore` one-liner that prevents an irreversible
   public-repo leak is the highest value-per-second item in the set.
3. **Order the test legs so the shared builder (T-5) lands before the
   parametrizations (T-3/T-4).** The hoisted `conftest.py` builder is what makes
   the parametrization clean.
4. **Bound the test consolidation.** T-3/T-4 must parametrize only the shared
   observable and preserve the behavior-specific tests — do not collapse coverage.
5. **Firmware-empirical assumptions** (`QUEUE_PARENT_ID="A:TRACKS"`,
   `play_from_queue` queue-preservation, 1-based `playlist_position`) are now
   well-commented audit trails — no action, but they remain the highest-risk
   assumptions on a future Sonos firmware update. A live-hardware **behavior
   test** for the reap-survival + say-resume flow would be the right regression
   gate if reliable hardware access becomes available to an automated executor
   (noted in the Mission-02 Flight-2 debrief); deferred here because it is
   HAT-only today.

  suite (~1,740 lines) — that agent both discharged four stale action items and
  surfaced the new overlap findings, which a combined Inspector might have
  under-weighted.
- **The Test Inspector corrected a briefed premise** — the SoundHelix SPOF is 2
  of 4 scripts, not 4 of 4. Worth carrying as a reminder that debrief-summarized
  debt should be re-verified, not transcribed.
- **No roundtable needed again.** The Architect verified findings against the
  live tree and the only disagreement (I-1 severity) was a downgrade the
  Inspector's own evidence already supported. For a project this small and this
  well-documented, the assess-then-verify pass is sufficient.
