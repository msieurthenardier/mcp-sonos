# Maintenance Report: 2026-05-18

**Date**: 2026-05-18
**Assessment**: Maintenance Required

## Inspection Scope

Full-codebase baseline inspection — the project's first routine-maintenance run, ad-hoc (no prior mission completed under Flight Control). Inspected by a two-agent wave: Inspector (categories 1–7 + 9) and Security Reviewer (deep manual). Architect (Opus) assessed severity; no roundtable cycle was needed — both reviewers calibrated reasonably for the documented trusted-LAN threat model.

**Delegation plan executed**: Single wave, two concurrent agents. Project is small (~9 main source files + 7 historical files in `poc/`); partitioning was unnecessary.

## Executive Summary

mcp-sonos is a small, well-maintained single-developer project with disciplined documentation and a coherent trusted-LAN threat model. The inspection surfaced one real correctness bug (`playlists.py:380` AttributeError on every external playlist takeover), one capability that exceeds the documented threat model (`play_file` can stage arbitrary readable files onto the LAN-public audio host), and a cluster of low-effort documentation and config-hardening items. Nothing blocks further work; a focused maintenance pass clears the meaningful items.

## Known Debt Carried Forward

None — this is the project's first inspection. The audio host LAN bind (F9) and TTS cache growth (F21) are documented in CLAUDE.md/README as accepted within the trusted-LAN threat model and on the roadmap; not "carried forward" in the debrief sense.

## Recommendations

1. **Land the maintenance mission shortly.** Flight 1 alone clears the real bug and the threat-model-crossing capability in ~90 minutes — high value per minute.
2. **Treat F7 (test scaffolding) as its own focused flight.** It needs a DI refactor (`server.py:33`) before unit tests can run cleanly. After F7 lands, F1's fix should grow a regression test.
3. **Pin Piper voice hashes (F8) before the project gets used in any less-trusted environment** than the maintainer's home LAN. The supply-chain attack chain (TLS subversion → poisoned ONNX → onnxruntime CVE → RCE on host) is improbable on a hobbyist LAN but real.
4. **Don't widen scope.** The cluster of `Pass` findings (broad `except` clauses, AudioHost.stop wiring, transitive deps, etc.) is genuinely not worth churn. The architect calibrated correctly.

## Delegation Effectiveness Notes

- **Single-wave two-agent partition was the right call.** Project is small; further partitioning would have wasted agent overhead. No agent hit context limits.
- **Architect's "no roundtable needed" decision saved a cycle.** Both reviewers were well-calibrated. For future inspections of larger projects, watch for false alarms in the audio-host directory-listing class of finding — easy to over-report when the project explicitly documents acceptance.
- **Inspector caught F1 (the real bug) and F4b (the doc-as-trap contradicting CLAUDE.md) — two findings that would be invisible to tooling-only inspection.** Reading CLAUDE.md before code paid for itself.
- **Security Reviewer's F2 was the highest-leverage finding** — the Inspector flagged the same surface as "play_file should validate" but Security elevated it from hygiene to threat-model-crossing with the prompt-injection framing.
